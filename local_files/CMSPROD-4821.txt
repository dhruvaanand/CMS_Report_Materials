================================================================================
RESOLUTION
================================================================================

Root Cause:
  CouchDB disk partition (/var/lib/couchdb) at T2_US_MIT reached 100% capacity
  due to accumulation of stale view index files following WMAgent design document
  updates. The old indexes were not cleaned up automatically.

Impact:
  WMAgent in Draining state for ~5h 47min (05:55 - 11:47 UTC).
  Approximately 12,400 job units delayed. Two TTbar NanoAOD workflows delayed.
  No data loss. No jobs failed due to this incident.

Fix Applied:
  1. Site admin cleared 218GB of stale CouchDB view indexes
  2. CouchDB service restarted
  3. WMAgent manually reconfigured via REST API to exit Draining state

Time to Resolution: 3h 15m from ticket creation (5h 47m from actual start)

Follow-up Actions:
  - CMSPROD-4822: Implement automated CouchDB index cleanup cron job at T2 sites
  - Consider adding CouchDB disk utilization to WMStats alerting dashboard
  - Document manual WMAgent reconfiguration procedure in CompOps runbook
