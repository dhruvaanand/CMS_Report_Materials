DESCRIPTION
================================================================================

Observing mass transfer failures between T1_DE_KIT (source) and T2_CH_CERN
(destination) starting around 13:02 UTC today. 

Rucio ConMon dashboard shows:
  - 8,412 files stuck in SUBMITTED state on T1_DE_KIT -> T2_CH_CERN link
  - Failure rate for this link: 97% over last 2 hours
  - Estimated blocked data volume: ~2.3 TB

FTS monitoring (https://fts3.cern.ch/ftsmon/) shows all transfers on this
link failing with:
  [GRIDFTP] 530 Login incorrect

Affected datasets include NanoAOD samples needed for ongoing BPH analysis
and several re-reco output datasets being staged for tape archival.

Transfers from T1_DE_KIT to other destinations (T1_ES_PIC, T2_DE_DESY)
appear unaffected. Transfers from other sources to T2_CH_CERN also
unaffected. Issue is specific to this site pair.

[S. Bhattacharya | 2025-12-02 14:22 UTC]
Cross-checked FTS job details. Error is consistently:
  Type: GRIDFTP
  Subtype: DESTINATION
  Reason: 530 Login incorrect (user/proxy certificate not accepted)

The 530 error code in GridFTP means authentication failure - the server
refused the presented credential. Since transfers from KIT to other sites
work fine, the KIT FTS service itself is running. The credential being
rejected must be specific to the KIT->CERN path, suggesting the proxy
used by KIT's FTS for CERN storage authentication has expired.

Tested manually from lxplus:
  $ gfal-ls gsiftp://eoscms.cern.ch/eos/cms/store/test/ \
      --cert /tmp/x509_kit_proxy
  gfal-ls error 13 (Permission denied): Permission denied (proxy expired 
  or not trusted by endpoint)

Confirmed: the grid proxy certificate being used by T1_DE_KIT's FTS 
service for authentication to CERN storage has expired.

================================================================================
RESOLUTION
================================================================================

Root Cause:
  MyProxy credential renewal cron job at T1_DE_KIT was silently failing for
  6 days due to a hostname change in the MyProxy server (myproxy.kit.de ->
  myproxy2.kit.de). The credential expired at 13:02 UTC, causing all FTS
  transfers from KIT to CERN to fail with GridFTP 530 authentication error.

Impact:
  8,412 files (~2.3 TB) blocked for approximately 5.5 hours.
  BPH NanoAOD staging delayed. Re-reco tape archival delayed.
  No data loss. All files eventually transferred successfully.

Fix Applied:
  1. KIT admin manually renewed MyProxy credential
  2. KIT admin updated cron job to use new MyProxy server hostname
  3. Stuck Rucio rules re-evaluated via rucio-admin
  4. Transfers completed normally

Time to Resolution: 5h 28m

Follow-up:
  - KIT will add monitoring for myproxy credential expiry (alert 24h before)
  - CompOps to add "check myproxy expiry" to transfer failure triage checklist
  - Consider adding credential validity check to Rucio ConMon alerts
